Module 1 Definitions & Concepts of Integration

What is Software Integration?

Definition of Software IntegrationDefinition▼

Software Integration refers to techniques for combining existing software components to form a complete system. It includes at least one existing software system developed in the past with no plan for symmetric integration with another component in the future.

Application Integration deals with integration approaches and techniques that connect different components of IT infrastructure — people, applications, platforms, and databases — to enable secure, intra and inter application collaboration.

Application IntegrationDefinition▼

Application Integration assists in unlimited sharing of data and business processes among any connected applications or data structures.

It is a process of connecting multiple, independently developed applications using incompatible technologies into a single organization-wide system with information flowing seamlessly among the integrated systems or components.

In short: Integration is the process of combining separate applications into cooperating collections of applications.

Types of Software Integration

4 Types of Software IntegrationTypes▼
TypeDescription
1. System IntegrationGiven two or more systems, subsystems or components — each functioning properly — they are combined to satisfy the combined requirements within a newly formed environment.
2. Functional / Technology IntegrationA software system already functioning properly must be updated by integrating a new function or technology while preserving the original functionality.
3. Incremental EngineeringA system is developed and delivered using available technologies; additional functions can later be integrated. The key challenge is designing the system with future integration in mind.
4. ModificationAn existing, properly functioning software system must be decomposed and reintegrated to carry out a modification.

Why Integration is Needed

5 Reasons for IntegrationBenefits▼
1
Technology Change: Technology change affects all layers; legacy systems do not go away easily.
2
Organizational Architecture: The architecture of an organization's information system depends on constraints related to technology AND the organization itself.
3
Departmental Silos: Within an enterprise, each department may have its own IT infrastructure maintained independently. Integrating them may bring additional value.
4
Zero Latency: Integration assists in forming a Zero Latency Organization — when all functions work with the same up-to-date information, latency between applications is eliminated.
5
New Applications: Updating and integrating applications is possible whenever required. New apps can be created by integrating real-time data from different parts of the enterprise.
Module 2 Integration Models & Methods

Integration Models

3 Integration ModelsModels▼
ModelDescription
1. Presentation IntegrationAllows integration of new software through the existing presentation (UI) of legacy software. Typically used to create a new user interface or integrate with other applications.
2. Data IntegrationAllows integration of software through access to the data that is created, managed, and stored by the software — typically for reusing or synchronizing data across applications.
3. Functional IntegrationAllows integration of software for the purpose of invoking existing functionality from other new or existing applications. Integration is done through software interfaces.

4 Methods in Application Integration

File TransferMethod▼

One application writes a file that another later reads. The applications must agree on the filename and location, the format of the file, the timing of when it will be written and read, and who will delete the file.

Shared DatabaseMethod▼

Multiple applications share the same database schema, located in a single physical database. Because there is no duplicate data storage, no data has to be transferred from one application to another.

Remote Procedure InvocationMethod▼

One application exposes some of its functionality so it can be accessed remotely by other applications as a remote procedure. Communication occurs in real-time and synchronously.

MessagingMethod▼

One application publishes a message to a common message channel. Other applications can read the message from the channel at a later time. Applications must agree on a channel as well as message format.

Challenges in Integration

5 Challenges of IntegrationChallenges▼
ChallengeExplanation
1. Network UnreliabilityIntegration solutions must transport data across networks. Unlike single-computer processes, distributed computing must deal with a much larger set of possible problems.
2. Distributed SystemsIn the worst case, every application runs on a completely separate environment with no shared database. Message-based communication is the only possibility to exchange information.
3. Slow NetworksSending data across a network is multiple orders of magnitude slower than local method calls. Designing a distributed solution like a single application could have disastrous performance implications.
4. HeterogeneityIntegration must transmit information between systems using different programming languages, operating platforms, and data formats. Each application implements its own data model, so mapping and transformations are required.
5. Modification ComplexityApplications change over time. Systems need loose coupling to minimize dependencies. Integrated systems are maintained independently and upgraded at different times.
Module 3 Integration Architecture & SOA

Integration Architecture

What is Integration Architecture?Architecture▼

Integration Architecture refers to the design and structure of systems and technologies that enable the seamless exchange of data and functionality between different applications, systems, and components within an organization or across multiple organizations.

It is a software architecture developed to facilitate the integration of IT components. Modern application architecture means companies use a wide variety of applications with different tasks — Integration Architecture ensures communication between these applications.

Benefits of Integration ArchitectureBenefits▼
BenefitDescription
1. Efficiency IncreasesCreates a control mechanism over daily processes. Employees can use all needed applications from a single entry point, doing more work in less time.
2. Reliable & Accurate DataNo disconnection; data is kept up-to-date in all components. The security of the data is not compromised.
3. Faster Decision MakingData is no longer dispersed into repositories. You can see all information as a whole, which is vital for faster decisions.
4. Reduces CostsReplacing all broken parts with a single integrated system protects the business from excessive financial expenses.

SOA – Service-Oriented Architecture

What is SOA?Architecture▼

SOA (Service-Oriented Architecture) is a design approach for building software systems composed of modular and interoperable services. It organizes software components into reusable and loosely coupled services that can be easily combined to create new applications and support business processes.

SOA provides ways to quickly integrate new applications and enhance collaboration between IT and business processes.

  • Creates reusable code — reduces time spent in project development.
  • Essential for scalability — helps scale your business to meet customer needs.
  • Increases interaction — produces solutions to prevent communication breakdowns between departments.
7 Characteristics & Principles of SOAPrinciples▼
PrincipleDescription
1. ServicesFunctionality is encapsulated as services — self-contained, modular components that expose well-defined interfaces and can be invoked over a network.
2. Loose CouplingServices can evolve independently without affecting each other. They communicate through standardized interfaces, enabling easy replacement or extension.
3. Service ContractsServices communicate using well-defined contracts specifying the interface (operations, parameters, return values) and any associated policies or constraints.
4. Service CompositionServices can be combined and orchestrated to deliver complex functionalities. Composition can be done dynamically, allowing for flexible and adaptable systems.
5. Service DiscoveryServices can be located, selected, and invoked at runtime based on their capabilities and availability. Service registries assist in discovery and management.
6. Scalability & ReusabilityServices can be scaled independently to handle load. Reusability is achieved by designing services as modular and generic components serving multiple applications.
7. Service SecuritySOA provides mechanisms for authentication, access control, and data protection enforced at the service level to ensure confidentiality, integrity, and availability.
Module 4 Middleware & Its Types

What is Middleware?

Middleware Definition & AdvantagesDefinition▼

Middleware is a layer of software that enables interaction and transmission of information between assorted applications and services that make up a complex system, providing a uniform user experience for clients.

In web development (e.g., Express.js), middleware gets executed after the server receives the request and before the controller actions send the response.

Advantages of Middleware:

  • Can process request objects multiple times before the server works on them.
  • Used to add logging and authentication functionality.
  • Improves client-side rendering performance.
  • Used for setting specific HTTP headers.
  • Helps for optimization and better performance.

Types of Middleware (14 Types)

1. Database MiddlewareType▼

The most common middleware. Enables access and interaction with different database gateways. Generic term for software infrastructure that supports: interoperability between software components, improved database service (performance scaling/fault tolerance), and adaptation to workloads.

2. Message-Oriented Middleware (MOM)Type▼

Allows software applications across multiple OS and networking protocols to receive and send messages to each other. Provides asynchronous inter-application communication. Message queues store messages on the MOM platform. Queues act as a central component for implementing asynchronous interaction.

3. PortalsType▼

Enterprises use portal middleware to facilitate interactions between client-facing systems and backend systems. Allows interaction between a client device and back-end systems (e.g., logging into an intranet or content management system).

4. Enterprise Application Integration (EAI)Type▼

A virtual layer that connects applications, data, processes, and services, irrespective of where they are located (cloud or on-premise) and what technology they are based on. Translates data and commands from one application format into another.

5. Transactional MiddlewareType▼

Ensures that transactions between heterogeneous components go through all necessary steps to reach completion. Includes TP monitors and application servers. Good fit with EAI because the architecture provides a centralized server able to process information.

6. Content MiddlewareType▼

Used to abstract specific content similar to publish and subscribe models. It includes database middleware, application server middleware, message-oriented middleware, web middleware, and transaction processing monitors.

7. Remote Procedure Call (RPC)Type▼

Allows one application to trigger a function in another application whether they reside in the same network or not. Makes an application functional across multiple platforms. Mostly used to initiate and process data transfers where both client and server are online simultaneously.

8. Device MiddlewareType▼

Contains a specific set of capabilities to integrate with or build applications on specific devices. Usually used to build mobile applications. Embedded middleware enables communication between embedded applications, operating systems, and applications.

9. Object Request Broker (ORB)Type▼

Handles requests from one application to another without letting the applications worry about where each is hosted. Provides a framework for clients to locate objects in the network and call operations on those objects as though they were local, providing location transparency.

10. Platform MiddlewareType▼

Allows business logic to be seated in any platform (OS or hardware), including web servers, application servers, hosting environments, or containers. Supports application development through managed tools and resources.

Examples: Runtime Environments (e.g., Java Runtime Environment), Web Servers, Content Management Systems (e.g., WordPress).

11. Application Server MiddlewareType▼

A framework that allows the creation, deployment, and maintenance of enterprise applications within a system. Provides distributed computing infrastructure, scalable and efficient application processing, and security services.

12. Web MiddlewareType▼

Allows companies to integrate more easily with individual backend systems. Works like a client-server architecture where the web services application is the client and the middleware is the server that provides services to the client.

13. Robotics MiddlewareType▼

Probably the most widespread middleware for robotics, representing the primary means for component integration in that domain. Simplifies the integration of robotic hardware, firmware, and software, irrespective of manufacturer and location.

14. Cloud MiddlewareType▼

Sits between the operating systems behind the cloud and the cloud users, providing a remote platform to create, maintain, and communicate with hosted applications and data. Always accessible to users in the form of a remote software platform for communication or management of data.

Module 5 Web Services & Middleware (IT 223)

What is a Web Service?

Web Service – DefinitionDefinition▼

Web Services are a method of communication between two electronic devices over the World Wide Web (WWW) network. They allow applications to communicate with each other and exchange data in a standardized format, such as XML or JSON (JavaScript Object Notation).

Web Services can be accessed by various clients including web browsers, mobile devices, and desktop applications, making them a popular choice for building interconnected systems.

They provide a standardized and platform-independent way for applications to communicate and share data over the internet, enabling integration and interoperability across diverse systems and environments.

Functions of Web Services

5 Functions of Web ServicesFunctions▼
1
Accessible via the internet or intranet networks.
2
Uses a standardized XML messaging protocol.
3
Independent of operating system or programming language.
4
Self-describing using the XML standard.
5
Locatable using a simple location approach.

Components of Web Services

1. SOAP – Simple Object Access ProtocolComponent▼

SOAP is a transport-independent messaging protocol built on sending XML data in the form of SOAP Messages. An XML document is attached to each message.

Only the structure of the XML document (not the content) follows a pattern. Everything is sent through HTTP, the standard web protocol.

2. UDDI – Universal Description, Discovery, and IntegrationComponent▼

UDDI is a standard for specifying, publishing, and discovering a service provider's online services. It provides a repository where WSDL files can be hosted so client applications can discover and learn about available web services.

The client application has full access to UDDI, which serves as a database for all WSDL files.

3. WSDL – Web Services Description LanguageComponent▼

WSDL is an XML-based file that explains what the web service does to the client application. It tells the client both where the web service is located and how to use it.

The client invoking a web service must know: (1) the location of the web service and (2) what the service does in order to invoke the correct one.

How Does Web Service Work?

5 Steps: How Web Service WorksProcess▼
StepDescription
1. Service ProviderThe entity that hosts the web service. Exposes functionalities or resources that other applications can access over the internet. Described in WSDL format.
2. Service ConsumerThe application or system that uses the web service's functionalities. Communicates with the provider to send requests and receive responses.
3. Request & ResponseCommunication happens over HTTP or HTTPS. The consumer sends a request specifying the desired operation and any necessary parameters.
4. Data ExchangeXML (used in SOAP) for structured/formal communication; JSON is lightweight and commonly used in RESTful web services.
5. InteroperabilityDifferent applications and systems can communicate regardless of the technologies they are built with, as long as both sides follow agreed-upon standards and protocols.

Features / Characteristics of Web Services

6 Features of Web ServicesCharacteristics▼
FeatureExplanation
1. XML BasedUses XML for information representation and data transportation. No need for networking, OS, or platform binding. Web service-based apps are highly interoperable at the middle level.
2. Loosely CoupledThe consumer is not necessarily directly linked to the provider. The provider's interface can change without impacting the consumer's ability to interact. (Contrast: tightly coupled systems require both sides to update together.)
3. Synchronous or AsynchronousSynchronous: client waits for the service to finish. Asynchronous: client invokes a task and continues with other tasks while waiting.
4. Coarse-GrainedMultiple fine-grained methods are combined into a single rough-grained service consumed by a buyer or service.
5. Supports Remote Procedural Call (RPC)Consumers can use XML-based protocol to call procedures, functions, and methods on remote objects.
6. Supports Document ExchangesXML's simple approach to data communication enables exchange of simple or complex documents, aiding corporate reconciliation.

Advantages of Web Services

5 Advantages of Web ServicesAdvantages▼
1
Business Functions over the Internet: Web services deliver functionality via HTTP, making them accessible from anywhere. All apps are now internet-accessible, making web services increasingly valuable.
2
Interoperability: Diverse apps can communicate and exchange information. A .NET application can communicate with Java web services and vice versa — platform and language independence.
3
Low-Cost Communication: Uses SOAP over HTTP, so existing internet connections can implement them. Also supports FTP and other reliable transport protocols.
4
Standard Protocol: All four layers of the web services protocol stack (Service Transport, XML Messaging, Service Description, Service Discovery) use well-defined protocols understood by everyone.
5
Reusability: A single web service can be used simultaneously by several different client applications.

Middleware – Deep Dive (IT 223 Slides)

Middleware – Definition (IT 223 Slides)Definition▼

Middleware refers to software that acts as an intermediary between different applications, systems, or components to facilitate communication, data management, and integration.

It sits between the operating system and the application, providing services and functionalities that help in building distributed systems, connecting heterogeneous components, and abstracting complexity.

6 Functionalities of MiddlewareFunctionalities▼
FunctionalityDescription
1. CommunicationFacilitates communication between distributed components by providing standardized protocols, messaging formats, and communication mechanisms. Enables reliable and efficient data exchange over networks.
2. IntegrationHelps integrate disparate systems, technologies, and platforms. Abstracts differences in data formats, protocols, and interfaces so applications work together seamlessly.
3. Data ManagementIncludes caching, transformation, and synchronization features. Provides services for accessing, storing, and processing data across distributed environments.
4. SecurityIncorporates mechanisms like authentication, authorization, encryption, and digital signatures to protect data and communications from unauthorized access, tampering, or interception.
5. Scalability & PerformanceOptimizes performance through load balancing, resource allocation, and distributed processing. Enables horizontal scaling across multiple nodes or servers.
6. Fault Tolerance & ReliabilityIncludes replication, redundancy, and failover mechanisms to ensure continuous operation and data integrity even when failures occur.
